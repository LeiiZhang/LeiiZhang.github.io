PhishEye: Live Monitoring of Sandboxed Phishing Kits（CCS 2016）

发表于 2019-09-01 | 分类于 CCF-A |
字数统计: 4.3k 字 | 阅读时长 ≈ 15 分钟

1 作者介绍

一作：Xiao Han
实验室：Orange Labs
Google Scholar：https://scholar.google.com/citations?user=Znrv-ysAAAAJ&hl=en&oi=sra



2 概述

2.1 摘要

  在本文中，我们提出了一种新的沙箱实时钓鱼工具包方法，可以完全保护受害者的隐私。基于 5 个月期间收集的数据，对钓鱼攻击、其机制、罪犯&受害者&参与该过程的安全社区的行为进行了全面真实的评估，绘制了第一个全面的钓鱼攻击图，从攻击者在受攻击的主机上安装和测试钓鱼页面开始，直到最后一次与真正的受害者和安全研究人员进行交互。我们的研究提供了对这种流行威胁的持续时间和有效性的精确测量，并讨论了通过监视数百次钓鱼活动所观察到的许多新的和有趣的方面。
  本文利用一个网络蜜罐来吸引真正的攻击者在一个 compromised 的网页应用中安装钓鱼工具。这是因为，根据 APWG 的全球网络钓鱼调查，承载钓鱼网页的域名中，有 71.4% 的域名为 compromised domains。本文提出了一种新的沙箱技术，旨在中和钓鱼工具包并在很长一段时间内保持其功能，严格预先服务于受害者的隐私，不干扰攻击过程，以确保攻击者可以破坏蜜罐，安装钓鱼工具，并进行功能测试，而不发现沙箱配置。之前的工作无法以白盒方式监视钓鱼工具包的生命周期。

2.2 本文贡献

（1）提出了一种新的方法，沙箱实时钓鱼工具包，完全保护终端用户的隐私；
（2）通过观察攻击者、受害者和安全研究人员与钓鱼页面的交互，首次重构了钓鱼工具包的整个生命周期；
（3）第一次测量黑名单服务对网络钓鱼页面的影响，从首次安装时开始（而不是从安全公司报告或发现的时间开始）。还讨论了使用收集的数据来迅速识别犯罪分子用来检索被盗信息的电子邮件地址的新技术。

3 背景

  蜜罐从来没有被用来研究受害者的行为，因为它们与安装的钓鱼网页连接和互动，需要考虑道德问题。

（1）服务器端
  钓鱼工具包旨在收集用户提交的数据。被窃取的数据可以存储在本地的蜜罐服务器上，直到被 phisher 检索到，也可以立即通过电子邮件或直接 HTTP 连接发送给 phisher。Moore 等人[29,30] 使用从易受攻击的web服务器上收集的公共可访问数据，包括 web 页面点击率的统计数据和攻击者已经收集并忘记从易受攻击的服务器上删除的凭证。在他们的研究发表之后，更多关于用户凭证的道德问题被暴露出来，其中大部分问题仍然没有得到解答。

[29] T. Moore and R. Clayton. Examining the impact of website take-down on phishing. In Anti-phishing working groups annual eCrime researchers summit, 2007.
[30] T. Moore and R. Clayton. Evil searching: Compromise and recompromise of internet hosts for phishing. In Financial Cryptography and Data Security. 2009.

（2）客户端
  除了上述关于如何防止信息存储在服务器上的道德问题外，客户端也存在道德问题。钓鱼工具可能通过将用户凭证发布到攻击者直接控制的其他恶意服务器来收集用户凭证。此外，97% 的钓鱼网页可以通过明文 HTTP 连接访问 [2]，从而暴露出受害者的敏感数据，以便通过网络窃听。因此，即使没有敏感数据驻留在远程服务器上，由于网络上用户凭证的明文共享，仍然存在相当大的风险。

[2] G. Aaron and R. Manning. APWG global phishing report 2014. http://apwg.org/download/document/ 245/APWG Global Phishing Report 2H 2014.pdf.

4 系统概述

  如图 2 所示，蜜罐系统包括 5 个模块。



（1）attacker tracking module
  攻击者跟踪模块负责将连接到蜜罐的攻击者与其他良性用户区分开来，这使能够应用不同的访问控制策略，使攻击者能够验证和测试他们的钓鱼工具包，同时防止受害者发布的任何敏感数据被攻击者捕获。该模块进一步跟踪使用攻击者 IP 地址及其关联用户代理的历史记录连接到蜜罐的所有攻击者，并将此信息作为输入提供给后续的保护和数据收集模块。

（2）client-side protection 和 server-side protection
  客户端和服务器端保护模块使用跟踪模块提供的攻击者列表作为输入，以识别潜在的受害者，并防止他们的数据被导出或从用户终端劫持。如果访问者不是攻击者，模块将检查每个传出的 HTTP 响应，并在每个页面中注入一个 HTML noscript 标记和一个静态 JavaScript 文件，该文件挂钩数据提交，并用虚假信息替换用户的数据。
  服务器端保护模块执行两个功能：后端数据随机化和恶意重定向破坏。在第一种情况下，它作为客户端保护之外的第二道防线。它在服务器端操作，并在传入数据传递到 web 应用程序之前用随机值替换这些数据。服务器端保护模块还拦截和阻止指向其他重定向网站的所有 HTTP、HTML 和 JavaScript 重定向，以确保蜜罐服务器不能用作更广泛的恶意重定向链中的垫脚石。它通过对服务器提供的每个 HTTP 响应的内容执行基于规则的静态分析来检测这种重定向，只允许重定向到托管在蜜罐中的其他页面。

（3）controlled SMTP module
  由于大多数网络钓鱼工具包通过电子邮件泄露被盗数据，蜜罐配置为丢弃传出的 SMTP 连接以防止攻击者发送垃圾邮件。我们还实现了一个 SMTP 模块，该模块允许跟踪模块识别的每个攻击者发送可配置数量的消息，以测试新安装的钓鱼工具包。

（4）data collection and processing module
  该模块进行数据收集和处理。

5 数据收集

  利用现有的蜜罐基础设施 [5] 作为基础来实现本文的系统。如图 2 所示，本文的体系结构由两个主要组件组成：（1）公共代理网关；（2）私有后端服务器（实现主蜜罐应用程序）。该网关托管在许多公共主机提供商上（包括 Amazon EC2，此前的研究显示 Amazon EC2 是攻击者的热门目标），代理服务器不承载任何内容，只充当反向代理，通过安全的 VPN 通道将所有 HTTP 连接转发到蜜罐服务器。出于安全原因，蜜罐发出的流量被丢弃在防火墙上，除了 DNS 查询和有限数量的经过验证的 SMTPS 连接。在蜜罐服务器上配置了 18 个易受攻击的 PHP 页面，这些页面可以被攻击者利用，并允许他们上传文件和执行 shell 命令，但攻击者无法修改这些 PHP 页面。数据收集模块定期检索蜜罐服务器上收集的数据，包括服务器访问日志和上传的文件（如 web shell、钓鱼工具、篡改页面、攻击工具和黑客工具）。

[5] D. Canali and D. Balzarotti. Behind the scenes of online attacks: an analysis of exploitation behaviors on the web. In Annual Network and Distributed System Security Symposium (NDSS), 2013.

6 钓鱼攻击全局图

  从 2015.9-2016.1，从蜜罐中共收集了 643 个钓鱼工具包，其中 474 个工具包被 471 个不同攻击者正确安装，剩下的工具包很可能是由开发机器人自动上传的，但从未被攻击者解包或配置。安装的钓鱼工具包针对 36 个不同的组织，主要是在线银行，也包括社交网络和电子商务门户网站。最常见的五个目标分别是 Paypal (375)、苹果 (26)、谷歌 (10)、Facebook (9) 和法国在线纳税系统 (6)。



  图 3 为在研究中观察到的所有钓鱼攻击的聚合时间表，Y 轴表示攻击的主要阶段（安装、测试、与第一个和最后一个受害者的交互、被流行的黑名单检测），X 轴表示自首次将网络钓鱼工具包上传到蜜罐后经过的时间。

6.1 攻击者的行为

  大多数攻击者只安装了一份钓鱼工具包，26% 的攻击者在多个子目录上安装了钓鱼工具包。安装完成后，70% 的攻击者访问了钓鱼页面，58% 的攻击者提交了假证书，以验证工具包是否正常运行。
  在工具包上传几分钟后，phishers 很快就把上传的工具包安装在了受损的 web 服务器上。测试阶段也在安装之后立即执行，然而，几分钟的延迟和不规则的模式表明，这是手工完成的。在测试阶段完成后，之后从未观察到来自该相同 IP 地址连接钓鱼页面。
  攻击者利用社会工程技术促使潜在的受害者连接到他们的网络钓鱼页面。在整个研究过程中，只观察到一个攻击者试图使用受攻击的机器发送钓鱼邮件（所有消息都被防火墙屏蔽了）。这表明，攻击者已经将危害公共服务器和安装钓鱼工具的过程与发送钓鱼消息的过程分离开来，通过使用不同基础设施使其更加健壮，并且降低了被检测的可能性。

6.2 受害者的行为

  经过过滤，统计了 2468 名受害者，他们连接了127个不同的钓鱼工具。我们的研究的优点是首次对钓鱼攻击的受害者数量进行了细致的评估，解决了过去导致高估钓鱼攻击受害者数量的两个主要限制：（1）在钓鱼页面首次出现在公共钓鱼黑名单之后，（第三方）访问者的数量出现了激增，而之前的研究没有将此与真正的受害者进行区分；（2）能够第一时间观察到受害者提交他们的信息到钓鱼页面。

6.3 钓鱼工具包的寿命

  只要新受害者连接到钓鱼网页，钓鱼工具包就是可以使用的。主要的挑战是很难捕捉到最后一个连接到钓鱼页面的受害者的确切时间。特别是，在网络钓鱼工具包被攻击者遗弃后的几天内，可能仍然观察到很少有用户连接到网络钓鱼工具包，从而导致高估了钓鱼工具包的生命周期。因此切断了分布的尾部，只考虑观察到的 90% 的受害者之间联系的时间间隔。如图 3 所示，第一个受害者捕获的时间，直到我们观察到 5% 的受害者连接到给定的钓鱼页面，最后一个受害者捕获的时间，直到我们观察到 95% 的受害者。使用该定义，在研究中观察到第一个受害者连接到网络钓鱼页面的平均时间为攻击者安装该页面后的 2 天，最后一个受害者（95% 的阈值）平均在 10 天后连接到网络钓鱼页面，即寿命估计为8天。

6.4 钓鱼黑名单的有效性

  主要目标是度量反钓鱼服务（GSB 和 PhishTank）将钓鱼 url 列入黑名单所需的时间，从相应的工具包首次被上传到蜜罐中开始。为了做到这一点，在我们的实验过程中，我们会定期检查，每天多次，这两个黑名单上的所有钓鱼网址都安装在我们的蜜罐。
  虽然几乎所有安装在蜜罐上的钓鱼网页（98%）都被这两个钓鱼黑名单正确检测并列入黑名单，但平均检测延迟为 12 天。分为两类：观察到受害者的在安装后平均 20 天内被列入黑名单，而没有受害者的工具包在安装后平均 10 天内被列入黑名单。如图 3 所示，检测延迟随钓鱼页面和攻击者使用的规避技术的不同而变化很大。更重要的是，75% 的受害者已经连接到相应的钓鱼页面后，62% 的钓鱼工具才被列入黑名单，结果表明反钓鱼服务没有有效的对某种类别的钓鱼攻击。在我们的研究中，他们确实对另外 27% 的网络钓鱼工具非常积极，甚至在 25% 的受害者尚未连接到网络钓鱼页面之前就将这些 url 列入了黑名单。

6.5 测量偏差

  在我们的研究中，我们评估了 PhishTank 数据集相对于我们的蜜罐中安装的471个钓鱼工具包的有效性。即使 375（约80%）个 PK 涉及PayPal，但只有一小部分（约1%）被报告，因而包含在 PhishTank 数据集中。这在一定程度上证实了我们数据集的多样性和代表性。因此，认为我们的测量是相对有代表性的。

7 案例分析

7.1 Dropping 技术

  如表 1 所示，绝大多数工具包使用电子邮件帐户向攻击者发送数据，很少有工具包直接将收集到的信息保存在服务器上，只有两个工具包使用 HTTP POST 请求将其发送到远程服务器。研究发现攻击者试验了一种新的 drop 技术，该技术旨在通过电子邮件从终端直接发送 HTML 表单中的信息。为了实现这一点，攻击者将 HTML 表单操作配置为将值直接发送到其电子邮件地址。尽管这种技术已经有文档记载，但这是第一次观察到钓鱼者在钓鱼工具包中实现它。然而这种方法非常不可靠，只适用于 Internet Explorer 和 Outlook Express 的特定组合。



  分析所有上传到蜜罐的工具包中包含的外部链接，发现 10 个工具包使用了直接从目标组织的 CDN 获取的资源，另外 98 个工具包中含有联系其他电脑的代码，这些电脑很可能在钓鱼者的控制之下。

7.3 受害者时间分布

  如图 5 所示，为 4 个最重要的钓鱼工具的受害者时间分布。



  观察到两种不同的分布形状：（1）一种歪斜的右分布，其中大多数受害者在工具包上载到蜜罐后的短时间内连接到钓鱼工具包；（2）一种双峰分布，其中两组受害者连接到两个不同时期的网络钓鱼工具包。图 5a、5b 和 5c 中所示的工具 1-3 属于第一类，在短时间内只有一组受害者，其次是极少数受害者的长尾。三个工具包中的钓鱼者可能只进行了一次钓鱼活动，之后他们放弃了这些工具包。与之前的工具包不同，图 5d 中的工具包 4 属于第二类，有两组不同的受害者。第一组在 24 小时内连接到网络钓鱼工具包，而第二组则在 70 小时（约 3 天）后到达，这可能是两种不同网络钓鱼活动的结果。
