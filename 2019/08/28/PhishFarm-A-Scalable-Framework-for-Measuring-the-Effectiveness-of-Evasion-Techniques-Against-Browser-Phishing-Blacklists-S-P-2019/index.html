PhishFarm: A Scalable Framework for Measuring the Effectiveness of Evasion Techniques Against Browser Phishing Blacklists（S&P 2019）

2 概述

2.1 摘要

  • 提出了 PhishFarm：一个可伸缩的框架，用于系统地测试反钓鱼实体和浏览器黑名单对攻击者逃避攻击的弹性(resilience)；
  • 使用 PhishFarm 部署了 2380 个 live 钓鱼站点(部署在新的、独特的和前所未见的 .com 域中)，每个站点使用基于真实钓鱼工具包的 6 种不同 HTTP 请求过滤器中的一种；
  • 向 10 个不同的反钓鱼实体报告了这些站点的子集，并测量了本地黑名单在主要 web 浏览器中的出现频率和时效性，以评估保护最终扩展到受害用户和组织的有效性；
  • 实验揭示了当前基础设施的缺陷；
  • 作者发现，代表真实世界攻击的简单伪装技术——包括基于地理位置、设备类型或 JavaScript 的攻击——平均可以有效地将黑名单的可能性降低 55% 以上；还发现黑名单在流行的移动浏览器（Chrome、Safari 和 Firefox）中并没有预期的功能，这使得这些浏览器的用户特别容易受到钓鱼攻击；
  • 在披露我们的发现后，反网络钓鱼实体现在能够更好地检测和缓解几种伪装技术（包括针对移动用户的技术），并且黑名单在桌面和移动平台之间也变得更加一致——但反网络钓鱼实体仍有工作要做，以确保用户得到充分保护；
  • PhishFarm 框架用来持续监测生态系统，可以扩展到测试未来恶意网站使用的最先进的规避技术。

2.2 文本贡献

（1）在现代桌面和移动浏览器中，服务器端请求过滤对黑名单覆盖率和及时性的影响的受控实证研究。
（2）一个可重用、自动化、可伸缩和可扩展的框架，用于执行本文的实验设计。
（3）识别当前反钓鱼生态系统中可操作的现实限制。
（4）在向反钓鱼实体披露后，加强了黑名单基础设施，包括 Chrome、Safari 和 Firefox 移动版本的钓鱼保护。

免费发布 PhishFarm 框架：https://phishfarm-project.com/



3 实验方法

  本文的主要研究目标是衡量伪装如何影响浏览器内钓鱼网站黑名单的发生和及时性。

3.1 Overview

  在较高的层面上，我们大规模部署自己的钓鱼网站（所有都是 PayPal.com 登录页面，都使用了新的、独特的和以前从未见过的 .com 域名，并且托管在横跨三大洲的不同IP地址集上），向反网络钓鱼实体报告其网址，并直接观察主要网络浏览器将其加入黑名单的时间。每次实验的过程如下：

1）选择要测试的特定反钓鱼实体；
2）部署大量新的，以前未见过的 PayPal 钓鱼站点与期望的伪装技术；
3）向实体报告这些站点；
4）衡量每个网站是否以及何时在主要网络浏览器中被列入黑名单。

  将实验分为两部分：对 10 个反钓鱼实体的初步测试（2017年年中）和对 5 个表现最好实体的全面后续测试（2018年年中）。

  • 目标 Web 浏览器：在选择要测试的浏览器时，努力使总市场份额最大化。考虑了所有主要的桌面和移动 Web 浏览器，如表 1 所示。


  • 过滤器类型：选择了一组基于高级伪装策略的请求过滤技术，这些策略是在最近对钓鱼工具包 .htaccess 文件[8] 的研究中发现的。表 3 总结了选择的过滤器。Filter A 作为对照组，期望这个组中的每个站点都能像其他站点一样尽快被列入黑名单；Filter B 研究纯移动网络钓鱼网站在多大程度上被列入黑名单；Filter C 和 D 特别关注受 GSB 保护的桌面浏览器，还包括地理定位，第二个动机是看看 GSB 以外的实体如何保护这个群体。Filter E 模拟典型的真实网络钓鱼工具包。 它基于最近的网络钓鱼工具包的大型数据集中的顶级 .htaccess 过滤器，旨在通过数百个 IP 地址，hostnames，referrers 和用 user agents 阻止反网络钓鱼实体；Filter F 只在客户端浏览器能执行 JavaScript 的情况下显示钓鱼内容，这可能会抵抗简单的基于脚本的 web 爬虫。


  • Tested Entities：表 2 总结了选择的实体。


  • Reporting：通过电子邮件或实体的网页界面向每个被测试的实体报告钓鱼站点的 URL。

3.2 初步测试

  每个服务器都有一个唯一的 IP 地址，托管在洛杉矶，纽约，多伦多，法兰克福，阿姆斯特丹或新加坡的主机数据中心之一。初步测试的每个批大小是 40 个钓鱼站点，总共有 400 个站点和 10 个实体。在每个批次中，6 个站点使用过滤器类型 A 和 F，而 7 个站点使用过滤器类型 B 到 E。表 4 为每批站点中 URL 类型和过滤器的分布。



3.3 负责任地揭露

  对初步测试的分析产生了若干安全建议，并立即向直接受影响的实体（即浏览器供应商、品牌本身和主要黑名单运营商）披露我们的发现。

3.4 Full-scale Tests

  在 2018 年 5 月注册了所需的域名，并在 7 月初启动了全面测试的顺序部署，重新测试了向其披露的所有实体。

3.5 样本大小选择

  对于 full tests，为每个实体选择 384 个钓鱼站点作为样本。我们的目标是通过单因素独立方差分析（one-way independent ANOVA）检验得到 0.95 的幂，显著性水平为 0.05，对于每一个实体，该检验能够识别出 6 个 cloaking filter 的平均黑名单速度存在显著性差异。假设 medium effect 大小 f 为 0.25，在每个实验中添加了 12 个站点（每个过滤器 2 个），以便在出现不可预见的技术困难时作为备份。

4 PhishFarm 实验框架



  PhishFarm 框架包括 5 个相互关联的组件，如图 1 所示：
  （1）批量域名注册和 DNS 设置；
  （2）无状态的云 web 服务器（显示钓鱼网站）；
  （3）API 和数据库管理配置；
  （4）client（通过 API 发出命令）；
  （5）monitoring system（定期检查哪些浏览器被列入钓鱼黑名单）。
总共为这些后端组件编写了 11,000 多行 PHP、Java 和 Python代码。

4.1 域名和DNS配置

  任何钓鱼网站的核心组件都是其 URL，由主机名和路径组成。我们的框架包含一个脚本，根据实验设计自动生成主机名和路径。然后，手工批量注册所需的域名，并将它们指向云托管提供商的域名服务器。最后，自动创建 DNS 记录，使每个实验的域名均匀地分布在我们控制的 IP 地址上。设置通配符 CNAME 记录，这样就可以在服务器端以编程方式配置每个钓鱼站点的子域。400 个初步域名的注册时间约为 15 分钟，1980 个域名的注册时间为 30 分钟，除了域名注册本身，不需要人工干预。

4.2 Research Client

  实现了一个跨平台的客户端应用程序来控制框架的服务器端组件。该 client 支持钓鱼站点和伪装技术的批量配置、实验的批量部署、自动化电子邮件报告、半自动网页报告、状态监视和数据分析。

4.3 服务器端组件

  服务器端组件基于动态配置显示实际的钓鱼站点，还负责记录请求数据，以便以后分析和监视黑名单状态。

（1）Central Database and API
  框架的核心是一个中央 API，它作为数据库的接口，提供钓鱼站点和系统状态。对于每个站点，数据库还维护诸如激活日期、报告日期和禁用日期；黑名单状态；网站和电子邮件 HTML 模板；服务器端和 JavaScript 请求过滤代码；访问日志。每当定义新站点或作为实验的一部分部署站点时，都通过客户机与这个 API 进行交互，所有进出 API 的通信都是加密的。

（2）主机基础设施（Hosting Infrastructure）
  主机基础设施由运行在 Apache 之上的自定义中间软件的 Ubuntu 云服务器组成。中间软件捕获所有请求，并支持动态伪装配置和增强的流量日志记录。每个服务器都从 API 获得其所有配置，这意味着可以根据测试需求灵活地选择服务器的数量（在我们的例子中是40个）。当任何服务器接收到 HTTP 请求时，服务器将根据 API 中的活动钓鱼站点列表检查请求 URL，处理任何 cloaking 规则，使用预期内容进行响应，并记录请求。为了快速处理传入的请求，在每个服务器上本地缓存系统状态，以最小化 API 查询。通过 HTTP 而不是 HTTPS 为所有的钓鱼站点提供服务，在每台服务器上本地托管所有资源（例如图像和 CSS）。

（3）监控基础设施（Monitoring Infrastructure）
  监视系统的目的是在细粒度上确定报告的钓鱼站点被列入黑名单之前所花费的时间，将监控系统配置为在每个目标浏览器中至少每 10 分钟访问一次实时钓鱼 URL。通过分析每个浏览器窗口的截图来检查每个 URL 的黑名单状态，检查每个图像中的主色是否与红色相似（浏览器的警告消息使用红色）。尽管这种基于图像的方法被证明是可靠的，并且可以完全自动化，但是它的伸缩性并不好，因为每个浏览器都需要单独使用一个系统的屏幕。
  为了满足可扩展性的要求，监控系统采用分布式架构，具有多个与 API 通信的自主节点，每个节点都是一个虚拟机，运行在一组更小的主机系统上。实验使用了三种类型的节点：Mac OS X 10.12 VMs with Chrome、Safari、Opera和Firefox；Windows 10 VMs with Edge and Internet Explorer (IE) 11；Windows 8.1 VMs with IE 11。每个节点通知它所支持的浏览器的API，然后等待由目标浏览器的一组 url 组成的命令，并实时向 API 报告监视结果。我们无法获得如此大量的移动设备，而当时的官方模拟器将强制禁用黑名单警告，因此每小时只测试移动浏览器一次。监视器使用匿名 VPN 服务（NordVPN），以绕过任何机构和 ISP 级别的嗅探。

5 评估



  表 5 显示了我们的钓鱼站点上的爬虫和黑名单活动的细分。 总体而言，带有 cloaking 的网站比没有 cloaking 的网站更慢，更不可能被列入黑名单。在 full tests 中，只有 23.0％ 的 cloaking 网站最终被列入至少一个浏览器的黑名单，远低于 49.4％ 没有 cloaking 的网站被加入黑名单。cloaking 还将平均黑名单时间从 126 分钟减少到 238 分钟。
  图 2 和图 3 绘制了随着时间推移被列入黑名单的站点的累计百分比，这些站点分别由每种浏览器或每种伪装技术划分。最后，我们分析网络流量日志，以观察每个反钓鱼实体的独特行为。

5.1 Crawling Behavior

  在我们在初步和全面测试期间启动的所有站点中，大多数（81.9%）看到了来自 web 爬虫程序的请求，其中大多数站点（66.0%）至少成功检索一次（即在使用时绕过伪装）。在 full 实验中，只有被爬的站点最终被列入了黑名单。

5.2 Entity Scoring

  对于每个实体测试，设 U 为报告给实体的钓鱼 URL 集，B 为被监控的浏览器集，T 是一组观察到的黑名单时间，F为所使用的 cloaking filter 集，MS 表示全球浏览器市场份额。此外，定义了函数 $accessible(b, f)$ 的值为 true，当且仅当一个带有过滤器 f 的钓鱼网站被设计为在浏览器 b 中可访问时。



  对于 $B \times U$ 中的每一个 URL-browser 组合，根据公式 1，我们定义了一个标准化的性能评分 $S_{URL_b}$，范围为 [0,1]，其中 0 表示没有黑名单，1 表示相对于报告时间立即列入黑名单。分数在 72 小时的观察窗内线性递减（例如，在 36 小时后被列入黑名单的站点的 $S_{URL_b} = 0.5$）。根据公式 2，我们将每个浏览器-过滤器组合的所有 URL-browser 得分的平均值作为 $S_{bf}$。
  进一步聚合 $S_{bf}$ 分数，总结每个实体对每个浏览器的保护。浏览器得分 $S_b$，如公式 3 所示，是给定浏览器 b 的所有 $S_{bf}$ 的平均值，但仅限于 b 中可访问的过滤器 f。为了评估每个 cloaking filte 的黑名单，将 $PB_f$ 报告为每个过滤器在至少一个浏览器中被列入黑名单的站点的原始比例。注意，$PB_f$ 总是大于或等于任何 $S_{bf}$，因为前者不会因为时效性差而减少。我们还报告了 $TB_f$，即每个过滤器 f 进入黑名单的平均时间（以分钟为单位）。
  为了获取每个实体的整体实际性能，按照公式 4 对所有 $S_b$（按每个浏览器的市场份额加权）进行平均，以产生S。分数 S 允许我们有效地比较实体之间的性能，并且将 有助于模拟 PhishFarm 未来部署的长期趋势。还包括所有已爬网站点的比例 C，以说明实体的响应工作量。
  表 VI 列出了所有实体在 full tests 中的综合得分。大量的 0 或接近 0 的得分令人失望，并代表了我们在下面几节中讨论的多个生态系统弱点。preliminary tests 得分见附录二表八。注意，由于 Chrome、Firefox 和 Safari 在所有实验中显示的分数几乎相同，所以我们简化了表格，在 GSB 标题下报告各自的最高分数。

5.5 实体性能

（1）Google Safe Browsing
  由于 GSB 所保护的浏览器在市场上占有很高的份额，它是当今最具影响力的反钓鱼黑名单，在实验中都获得了最高分。GSB 的关键优势在于它的速度和覆盖范围，crawler 通常会在报告后几秒钟访问我们的站点。在 full tests 中报告的所有站点中，有 94.7% 被爬取，97% 没有使用 cloaking （Filter A）的站点最终被列入黑名单。Filter D 的黑名单具有可比性，Filter F在初步测试中得到改进。
  然而，GSB 与 Filter E 斗争，它阻止了 Google 抓取工具特有的主机名。 它还与 Filter C 斗争，它拒绝非美国的流量。 相应的 $PB_f$ 分数低的原因是，如果网络钓鱼站点上的初始爬虫失败，GSB 将放弃进一步的爬行尝试。最初的命中几乎总是来自一个可预测的非美国 IP 地址。 另一个弱点似乎是数据共享，因为我们向 GSB 报告的网站都没有被 Edge，IE 或 Opera 列入黑名单。

（2）Microsoft SmartScreen
  事实证明，SmartScreen 是唯一一个利用主动 URL 启发式将钓鱼网站列入黑名单的本地反钓鱼黑名单，这使得微软浏览器在初步测试中获得了高分。这些启发式主要由具有欺骗性域名的 url 触发。在初步测试中，Edge被证明受到了非常好的保护，得到最高的 $S_b$ 得分，0.87分，是所有浏览器中最高的。
  在 full tests 中，IE的性能较初步测试有所提高，与 Edge 的性能更加一致。令人惊讶的是，SmartScreen 更有可能将有 cloaking 的网站列入黑名单，这可能是由于使用了伪装分类(这是值得称赞的)，同时对我们的钓鱼报告的信任度较低。向 SmartScreen 的报告似乎没有显著影响任何非微软浏览器，该实体与 GSB 有类似的缺点。SmartScreen 也是对钓鱼报告反应最慢的实体之一，而且它的整体覆盖率很低，这是它的关键弱点。

（3）APWG
  在 full tests 中，APWG 是得分第二高的实体，并且显示了对所有浏览器的一致保护。由于改进来绕过 Filters C 和 E，APWG 报告的得分与初步测试相比有了显著提高，这使得 APWG 报告在 GSB 浏览器中将此类站点列入黑名单，而这是我们直接向 GSB 报告时没有实现的。APWG 还生成了我们测试的所有实体中最高级别的爬虫流量。APWG 在 full tests 中没有将任何带有 D 或 F 过滤器的站点列入黑名单。

（4）PhishTank
  PhishTank 是一个社区驱动的交流中心，允许人类志愿者识别钓鱼内容，它还利用爬虫器和合作伙伴的网络来帮助这项工作。在我们的初步测试中，由于它在 GSB 浏览器中被迅速列入黑名单，因此在性能上排名第二。在完整的测试中，只有 46.7% 的站点被爬行，最终被列入黑名单的站点非常少。尽管如此，PhishTank 还是产生了第二高的爬虫总流量。我们不知道其缺陷的原因，PhishTank 也没有回复我们的披露，我们怀疑 PhishTank 分类的手工性质可能是一个限制因素。

（5）PayPal
  在初步测试期间，PayPal 自己的滥用报告服务难以绕过 Filters D 和 F，但在完整的实验中克服了后者，同时保持了适度的黑名单。它在两次测试之间也改进了对 Opera 的保护。尽管我们在完整测试中报告了除了两个网站外的所有站点，但平均响应时间和浏览器保护最终还是很差。我们不能透露原因，但预计未来会因我们的发现而出现改善。
